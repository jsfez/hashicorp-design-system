# Plan Generation Prompt: Replace `--token-*` CSS Custom Property Usage with `--hds-*`

## Context

The Helios Design System (HDS) design tokens are generated by `packages/tokens` (a
style-dictionary build) and consumed as CSS custom properties throughout the
component library, the showcase app, the website, and downstream consumer apps.

Historically those tokens were emitted with a `--token-*` prefix
(e.g. `--token-color-foreground-strong`, `--token-tag-border-radius`). As part of a
multi-step naming-convention effort, the token build has already been changed to
emit an `--hds-*` prefix instead:

- **Step 1 (done):** Component-scoped CSS custom properties were renamed
  `--hds-*` → `--hds-var-*` to disambiguate them from tokens.
- **Step 2 (done):** `packages/tokens` was updated so the generated CSS/SCSS output
  uses the `hds` prefix instead of `token`. `packages/tokens/dist` now emits
  `--hds-*` custom properties (and `$hds-*` Sass variables). There are **no more
  `--token-*` names produced by the token build.**
- **Step 3 (this prompt):** Update every **consumption site** that still references
  the old `--token-*` names so they point at the new `--hds-*` names.

This step is therefore not a token redefinition — it is a mechanical find-and-align
of all the places that *read* tokens, so they match the new output of step 2.

### The two namespaces (for disambiguation)

- **`--token-*`** — the OLD generated token prefix. All remaining occurrences in
  consuming code are stale references that must be updated. **Target of this step.**
- **`--hds-var-*`** — component-scoped internal/JS-bridge/public-override CSS
  variables (renamed in step 1). **Never touched by this step.**
- **`--hds-*`** (bare, not `--hds-var-`) — the NEW token prefix produced by step 2,
  plus a small set of public-override API layout variables
  (`--hds-layout-flex-row-gap`, `--hds-layout-flex-column-gap`,
  `--hds-layout-grid-row-gap`, `--hds-layout-grid-column-gap`) and a historical
  `--hds-app-desktop-breakpoint` reference that only appears in prose. The rename
  produces names in this namespace.

---

## Goal

Produce a **reusable, Ember-codebase-agnostic automation** that replaces all
`--token-*` CSS custom property usages with `--hds-*` in any target Ember codebase
that consumes HDS design tokens.

The same prompt/tooling produced for this step must be reusable, as-is, against
**other HashiCorp Ember apps** that consume the HDS tokens. The HDS monorepo
(`packages/components`, `showcase`, `website`) is only the first pilot / validation
target and has no special status. All in-scope consumer codebases are Ember-based —
the automation does not need to support non-Ember stacks.

---

## Requirements and Constraints

### The rename rule
- **Pattern:** `--token-` → `--hds-` (prefix swap only).
- **The token name after the prefix is identical** — only the leading `--token-`
  segment changes. Example: `--token-color-foreground-strong` →
  `--hds-color-foreground-strong`.
- **Applies to:** every CSS custom property *usage* that names a token — almost
  always inside `var(--token-...)`, but also bare property references in inline
  styles, JS/TS string values, type definitions, and template attributes.
- **Does NOT apply to:** anything that is not prefixed `--token-`. The pattern only
  matches the `--token-` prefix, so `--hds-*` and `--hds-var-*` names are never
  touched.

> **Important — a second, non-CSS class of prefix coupling exists.** Beyond the
> `--token-*` CSS custom properties, some code couples to the bare `token-` prefix
> as a plain string/regex when reading a token's `name` field (which comes from the
> tokens package JSON). Step 2 already changed that `name` field to the `hds-`
> prefix, so these sites are **already silently broken** and the `--token-`
> replacement will never surface them (there is no leading `--`). These must be
> handled as a separate concern: the automation should **flag each candidate with a
> TODO marker comment** (rather than editing it), so the user can review and decide
> post-run — see the "Indirect prefix coupling" pattern in the pilot context and the
> dedicated plan step below.

### Safety constraints
- The replacement must be **purely mechanical** — no semantic changes, no logic
  changes, no token value changes, no re-mapping of token names.
- Because only the `--token-` prefix is matched and the token name is preserved
  verbatim, there is **no risk of collision** with other namespaces: the editable
  source in the pilot repo contains no bare `--hds-*` names (they exist only in
  excluded Markdown), and the `--hds-var-*` namespace (step 1 output) is disjoint
  and never matched.
- After the replacement, the token references at consumption sites must resolve to
  real custom properties emitted by `packages/tokens` (i.e. the renamed `--hds-*`
  name must exist in the generated token output).
- **Idempotent and repeatable.** The tooling will be run repeatedly during
  iteration: run → review the Git diff → improve the plan/tool → `git restore` the
  changed files → re-run. It must therefore be safe to run multiple times, assume a
  clean checkout as input, depend on no prior-run state, and produce a byte-for-byte
  identical diff on every run over the same source.

### Target environments for the automation

Target codebases are Ember projects (the HDS monorepo and Ember consumer apps built
with it). File types in scope for edits:
- CSS and SCSS (`.css`, `.scss`)
- JavaScript and TypeScript (`.js`, `.ts`)
- Glimmer single-file components (`.gts`)
- Handlebars templates (`.hbs`)

### Out of scope / excluded

- **All `dist/` build output** in every package/app — regenerated by `pnpm build`,
  never hand-edited.
- **Compiled CSS assets** such as
  `showcase/public/assets/styles/@hashicorp/design-system-components.css` and
  related vendored/compiled stylesheets — generated artifacts, not source.
- **Source maps** (`*.map`).
- **Markdown files** (`*.md`) — documentation, CHANGELOGs, version-history and
  release-notes references are intentionally **excluded**. Consumer codebases do
  not have Markdown files that consume HDS tokens, and the HDS repo's own docs will
  be updated manually and separately.
- **No changeset and no consumer migration guide** are required for this step. The
  change is technically non-breaking (token names are preserved; only the prefix
  moves in lock-step with the already-updated token output). Any changeset or
  consumer communication will be handled separately by the maintainers.

## Deliverables to Plan

> The exact automation shape is **left to the plan to analyze and recommend** — see
> "Approach analysis" below. The plan must justify its recommendation with the
> reusability requirement front of mind, because the chosen tooling will be re-run
> against other Ember consumer codebases.

### Intended usage and iteration workflow (design for this)

The tooling is developed and hardened against the HDS monorepo (the pilot) through
an iterative loop: generate a plan → review it → run the tool → inspect the Git diff
→ refine the plan/tool to cover any gap or incorrect edit → `git restore` the
modified files → re-run → repeat until the outcome is clean. Once the maintainer is
satisfied, the tooling is copied into a **separate Ember consumer repo** and run
there with minimal fine-tuning. The plan should therefore optimise for:
- **Portability** — maximise what works unchanged across repos; isolate the few
  things that legitimately differ (paths, exclusions) into configuration.
- **Reviewability** — produce clean, minimal diffs and a clear summary of what
  changed (and what was flagged for manual review) so each iteration is easy to
  audit in Git.
- **Repeatability** — see the idempotency constraint above; every re-run from a
  clean checkout must yield the same result.

**The plan is the durable, portable knowledge artifact.** On every iteration, the
AI must fold new learnings, edge cases, gotchas, and context discovered during
execution/review back into the plan itself (and the tooling's docs/config), rather
than leaving them only in chat. Nothing learned while hardening against the pilot
should be lost — the plan must stay self-contained enough that it, plus the tooling,
can be carried into the next repo and understood without replaying this
conversation.

The plan (and any knowledge captured during iteration) must be **persisted as one or
more files in the repository**, not held in memory or chat only — Markdown for
human-readable plan/documentation, and other formats only where strictly necessary
for execution (e.g. JSON or CSV for structured data the tooling itself consumes).

### Approach analysis (required first section of the plan)

The plan must open with a short analysis that weighs the pragmatic options and
recommends one, given that the tooling must be reusable across multiple Ember
codebases with a single invocation. Options to evaluate include, at minimum:

- A pure deterministic find-and-replace (e.g. a scripted CSS/SCSS + text transform)
  with an AI verification pass to catch leftovers.
- A deterministic transform for stylesheets plus a `packages/codemods`-style
  jscodeshift codemod for JS/TS/GTS/HBS, orchestrated together.
- An AI-driven replacement with deterministic verification as the guardrail.

For reference, step 1 (`--hds-*` → `--hds-var-*`) initially considered complex
codemod tooling but concluded that a well-scoped find-and-replace plus AI
verification was sufficient. Since step 3 is a strictly simpler, collision-free
prefix swap on token *usages* only, the plan should critically assess whether
heavyweight codemods are warranted or whether a leaner, more portable tool better
serves the "run it on many repos" goal.

### Expected artifacts (final shape decided by the analysis)

Whatever approach is chosen, the plan should produce:
- A reusable replacement tool/script that can be pointed at any Ember repo root and
  applied to the in-scope file types. The **reusable transform logic must be kept
  separate from repo-specific configuration** (target roots, in-scope extensions,
  exclusion globs) so the tool can be lifted into another consumer repo and adjusted
  by editing config only — not the core logic. Sensible HDS defaults may ship with
  it, but they must be overridable.
- A verification step that reports any remaining `--token-*` occurrences in source
  (excluding the excluded paths) and confirms the count reaches zero.
- If (and only if) the analysis concludes codemods are worthwhile, a
  `packages/codemods` codemod following that package's existing conventions.
- A reusable AI skill (or equivalent agent-instructions artifact) whose primary job
  is to verify the deterministic pass left no `--token-*` usages behind, and whose
  secondary job is to act as the driver in a repo where the deterministic tool
  cannot be run.

## Plan Structure Expected

The plan to be generated should include:

1. **Approach analysis and recommendation** (see above).
2. **Pre-conditions / audit step** — confirm the target repo's token build (or the
   installed `@hashicorp/design-system-tokens` version) already emits `--hds-*`, and
   count baseline `--token-*` occurrences in source so success can be measured.
3. **Implementation steps** for each chosen artifact, including:
   - File location: the AI execution should **propose** where to create/place the
     artifact and prompt the user to confirm or override it case by case, rather
     than hard-coding a location.
   - Input/output contract.
   - Exclusion globs (`**/dist/**`, compiled asset stylesheets, `*.map`, `*.md`).
   - Edge cases to handle: `var(--token-...)` in SCSS, `var(--token-...)` inside JS/TS
     string values (e.g. modifiers, theme objects), TypeScript custom-property type
     keys, GTS/HBS template attribute values (e.g. `@color="var(--token-...)"`),
     inline `style="--token-...: ..."`, test assertions (`hasStyle`, string-form
     style assertions), and **computed/interpolated token names** where the
     `--token-` prefix is literal but the suffix is built dynamically
     (e.g. Sass `var(--token-color-#{$product}-brand)`). These are still covered by
     the mechanical prefix swap — only the literal `--token-` head is replaced and
     the interpolated suffix is left intact — so they must **not** be skipped.
4. **Execution order** — for the HDS pilot: SCSS sources
   (`packages/components/src/styles/`) → component GTS/TS sources
   (`packages/components/src/`) → showcase app + tests (`showcase/app/`,
   `showcase/tests/`) → website templates/snippets (`website/`, `.hbs`/`.gts`/`.ts`
   only), then rebuild.
5. **Testing / validation strategy** — diff review, `pnpm build` in
   `packages/components`, and lint across the three folders (`pnpm lint`, and
   `pnpm lint:fix` where appropriate). Note explicitly that resolving a renamed
   `--hds-*` reference depends on the corresponding token existing in the built
   token output. The AI execution must **not run the test suite itself** (it is
   slow and may hang) — instead it should ask the user to run the tests and report
   results back, then act on those results.
6. **Verification / fallback strategy** — how the AI agent verifies zero remaining
   `--token-*` usages in source, and when it should act as the primary driver
   instead of the deterministic tool.
7. **Indirect prefix-coupling audit (annotate, don't auto-fix)** — a separate sweep
   for the bare `token-` prefix (not preceded by `--`) in `.js`/`.ts`/`.gts`,
   focused on code that reads a token's `name` field. Because step 2 already changed
   the token JSON `name` to the `hds-` prefix, any such comparison/strip is already
   broken. Rather than triaging these during the run, the automation should:
   - **leave the flagged line itself unchanged** (no auto-replacement — blind edits
     would corrupt false positives such as arbitrary identifiers containing
     `token-`),
   - **insert a TODO marker comment on the line above** each candidate, using a
     stable, greppable tag and the correct comment syntax for the file/region
     (e.g. `// 🚧 TODO [HDS-TOKEN-RENAMING]: verify manually — token-name prefix
     coupling` for JS/TS/GTS logic; `{{! ... }}` inside template regions),
   - **emit a summary report** listing every annotated file and line, so the set can
     also be reviewed and the markers removed in bulk once resolved,
   - document that the user reviews each marker post-run and decides case by case
     (fix — preferably by deriving the prefix from the tokens package — or delete the
     marker if it is a false positive).

The plan may keep the exact packaging of the AI layer as an implementation detail
but should state whether a reusable skill is preferable to a one-off prompt and
outline the minimum responsibilities that AI layer must cover. It may also recommend
phased or sub-agent execution if repository size or context limits make a
single-agent workflow impractical (the pilot's `packages/components` alone contains
~1,500 occurrences).

## HDS Monorepo: Pilot and Validation Context

The following facts were gathered by scanning `packages/components`, `showcase`, and
`website` in the HDS monorepo. Use this as the initial context so the automation
plan doesn't start from scratch — validate and stress-test the automation here first
before generalising guidance for other Ember consumer codebases.

### Scope Summary (source only; excludes `dist/**`, compiled `public/assets/**`, `*.map`, `*.md`)

| Folder | `--token-*` occurrences |
|---|---|
| `packages/components` | ~1,525 |
| `showcase` | ~78 |
| `website` | ~67 |

### Occurrences by file type (source only, all three folders)

| Extension | Occurrences |
|---|---|
| `.scss` | ~1,527 |
| `.ts` | ~64 |
| `.gts` | ~44 |
| `.js` | ~7 |
| `.hbs` | ~4 |
| `.md` | ~24 (EXCLUDED — not migrated) |

### Affected File Types and Usage Patterns

#### 1. SCSS files (`*.scss`) — the dominant case

Almost all usages are `var()` reads of a token:

```scss
.hds-tag {
  border-radius: var(--token-tag-border-radius);
  max-width: var(--token-tag-max-width);
  padding: var(--token-tag-padding-top) var(--token-tag-padding-right)
    var(--token-tag-padding-bottom) var(--token-tag-padding-left);
}
```

Some SCSS token names are **computed via Sass interpolation** — the `--token-`
prefix is literal but the suffix is built dynamically. The mechanical prefix swap
still applies (only the literal head changes; the `#{...}` suffix is untouched):

```scss
// packages/components/src/styles/components/icon-tile.scss
@each $product in $hds-icon-tile-colors-products {
  &.hds-icon-tile--color-#{$product} {
    color: var(--token-color-#{$product}-brand); // → var(--hds-color-#{$product}-brand)
  }
}
```

#### 2. TypeScript / JavaScript files (`*.ts`, `*.js`)

Token references embedded in string values — e.g. modifiers and code-editor themes:

```typescript
// packages/components/src/modifiers/hds-tooltip.ts
'var(--token-tooltip-max-width)'

// packages/components/src/modifiers/hds-code-editor/themes/hds-theme.ts
color: 'var(--token-code-block-foreground-color-primary)',
backgroundColor: 'var(--token-code-block-surface-color-primary)',
```

#### 3. GTS files (`*.gts`) — component source and showcase

Token references passed as component argument / attribute values:

```hbs
{{! packages/components/src/components/hds/application-state/header.gts }}
@color="var(--token-color-foreground-strong)"
```

```hbs
{{! showcase/app/components/.../base-elements.gts }}
@statusIconColor="var(--token-color-waypoint-brand)"
```

#### 4. HBS template files (`*.hbs`) — website code snippets

Token references inside documented code examples:

```hbs
<Hds::Icon @name="alert-circle" @color="var(--token-color-foreground-success)" />
```

#### 5. Test files (`*.gts` in `showcase/tests/`)

Token references appear both as live argument values and inside `hasStyle`
assertions / commented expectations. Note the co-existence with already-migrated
`--hds-var-*` names — the tool must change only the `--token-*` part:

```typescript
@statusIconColor="var(--token-color-waypoint-brand)"
// .hasStyle({'--hds-var-app-footer-status-icon-color': 'var(--token-color-waypoint-brand)'})
```

#### 6. Indirect prefix coupling (`*.js`, `*.ts`, `*.gts`) — NOT a `--token-*` usage

A distinct and easy-to-miss class: code that reads a token's `name` field (sourced
from `@hashicorp/design-system-tokens` JSON) and couples to the bare `token-` prefix
via a string or regex. These have **no leading `--`**, so the `--token-` replacement
never touches them — yet step 2 already changed the JSON `name` field to the `hds-`
prefix, so this logic is **already silently broken**.

Confirmed examples in the pilot:

```typescript
// showcase/app/components/page-carbonization/foundations/color/index.gts
return token.name?.replace(/^token-/, ''); // no longer strips anything (name is hds-*)

// website/app/components/doc/token-preview/index.gts
this.token.name.startsWith('token-typography')            // now always false
this.token.name.startsWith('token-typography-font-weight')

// website/app/components/doc/page/header/algolia-search/parts/htmlTemplatesItemPreview.js
tokenName.match(/^token-typography/)   // now never matches
tokenName.match(/^token-color/)
```

These require care rather than blind replacement:
- **True couplings** (compare/strip against the token-name prefix) must be fixed —
  ideally by deriving the prefix from the tokens package rather than re-hard-coding
  `hds-`, but at minimum by updating `token-` → `hds-` in the literal/regex.
- **False positives must be left untouched** — arbitrary identifiers that merely
  contain `token-` (e.g. Algolia index attribute keys `'token-name'`,
  `'token-type'`, `'token-value'`, `'token-group'`; component/partial/snippet names
  like `token-preview`, `token-import`, `token-as-component-argument`).

Because these cannot be safely auto-replaced, the automation should **not** try to
decide them inline. Instead it should **annotate each candidate with a TODO marker
comment** on the line above, so the user can review and make the judgement call
after the replacement flow completes (see the dedicated plan step). This class is
repo-specific: consumer codebases may or may not have any.

### Key structural facts (validated during research)

- The swap is a **pure prefix change**; token names are byte-for-byte identical
  after `--token-`/`--hds-`.
- **No Sass `$token-*` variables** exist in any of the three folders — token
  consumption is entirely via CSS custom properties. (Do not invent a `$token-*`
  rule.)
- **No collisions:** the editable (non-Markdown) source of the three folders
  contains no bare `--hds-*` names, and `--hds-var-*` is a disjoint namespace the
  rule never matches.
- The very large raw counts seen in unfiltered scans (tens of thousands in `.css`,
  thousands in `.map`) are entirely **generated/compiled assets** and must be
  excluded — real source scope is ~1,670 occurrences total.

### Files to EXCLUDE (do not edit)
- `**/dist/**` in every package/app (built output, regenerated by `pnpm build`).
- `showcase/public/assets/styles/@hashicorp/design-system-components.css` and other
  compiled/vendored CSS assets.
- `**/*.map` source maps.
- `**/*.md` Markdown (docs, CHANGELOGs, version-history, release-notes) — updated
  manually and separately.

## Additional Notes for Plan Author

- The HDS monorepo already has `packages/codemods` (jscodeshift-based). If the
  analysis concludes a codemod is warranted for JS/TS/GTS/HBS, it should follow the
  conventions there — but the analysis should first justify the codemod against the
  simpler find-and-replace alternative, given the reusability goal.
- The HDS SCSS instruction file at `.github/instructions/languages/scss.instructions.md`
  defines code-style constraints for SCSS edits.
- Do **not** touch Sass variables of any kind — this step is strictly about
  CSS custom property token usages (`--token-*` → `--hds-*`).
- Linting may be run freely (`pnpm lint`, `pnpm lint:fix`). The test suite must
  **not** be run by the automation/agent — it is slow and may hang; the agent
  should ask the user to run the tests and report back.
- The automation must accept an arbitrary Ember repo root, apply the in-scope file
  globs and the exclusion globs above, and finish with a verification that zero
  `--token-*` usages remain in source.
