## 1. Context & goal

The Helios Design System (HDS) design tokens are generated by `packages/tokens` (a style-dictionary build) and consumed as CSS custom properties throughout the component library, the showcase app, the website, and downstream consumer apps.

As part of **Project Solar**, the token set is undergoing a large **"carbonization"** refactor. Carbonization is an **umbrella effort** that bundles together several kinds of token change accumulated across the work ŌĆö both **mechanical** and **non-mechanical**:

- **Mechanical renames**, e.g. the `--token-*` ŌåÆ `--hds-*` prefix swap (see `copilot-plans/hds-tokens-replacement/`) and the `--hds-*` ŌåÆ `--hds-var-*` component-internal variable renaming (see `copilot-plans/hds-var-renaming/`) ŌĆö simple, rule-based transformations.
- **Non-mechanical refactors** ŌĆö tokens **renamed, removed, and added** in arbitrary ways (e.g. the semantic color namespace moved from `color-[semantic]` to `[semantic]-color`: `--token-color-border-primary` ŌåÆ `--hds-border-color-primary`).

**Verified during recon (see ┬¦3):** the pre set (`main`) is **entirely `--token-*`** (442 tokens) and the post set (working tree) is **entirely `--hds-*`** (801 tokens) ŌĆö so **every** pre token changes at minimum via the universal `--token-` ŌåÆ `--hds-` prefix swap, which is effectively **step 0** of carbonization. Of the 442 pre tokens, **242** need only the mechanical prefix swap (their prefix-swapped name exists verbatim in post), **200** need further inference (their prefix-swapped name is absent from post because of a non-mechanical refactor), and the post set additionally contains **559** names with no prefix-swapped pre counterpart (brand-new tokens plus rename targets). A practical consequence: because the prefix swap is universal, **no token is literally unchanged** ŌĆö so a na├»ve pre/post name diff would put the entire delta in one undifferentiated "removed" pile. Instead the map groups pre tokens by **how** they changed (see ┬¦4): `prefix-only` (242), the `prefix-plus-renaming__*` families (184), and `removed` with no successor (16); the 377 brand-new post tokens are `added`.

Because the non-mechanical part has **no single find-and-replace rule**, the overall migration requires an explicit oldŌåÆnew mapping. This tooling therefore treats carbonization as **one combined delta**: it diffs the pre- vs post-carbonization token **names** and captures whatever changed ŌĆö mechanical and non-mechanical alike ŌĆö in a single map. (Token *values* also changed in places, but values are irrelevant to this migration, which only rewrites token **names**.)

### Goal

Produce reusable tooling that:

1. **Phase A ŌĆö mapping generation:** diffs the pre- and post-carbonization token **name** sets and produces a proposed oldŌåÆnew **token map** (see ┬¦4). The map is an editable JSON that serves triple duty: input to Phase B, human reference, and a basis for documentation/migration-guide generation.
2. **Phase B ŌĆö migration:** takes the **human-confirmed** token map and mechanically rewrites every stale token usage in consumer code to the new name, flags anything with no mapping for manual review, and verifies zero stale token names remain.

The two phases have very different reuse profiles:

- **Phase A runs once, on this monorepo** ŌĆö it is the only place with all the inputs needed to build the map (the `main` baseline, the changesets, the token source). It does **not** need to be portable. Running it here is what produces the two artifacts that then travel onward: (1) the confirmed **token map JSON**, and (2) a validated **Phase B** migration tool hardened against this repo before it is pointed at other codebases.
- **Phase B is the reusable, config-driven tool** (like `copilot-plans/hds-tokens-verify/tooling/verify-tokens.mjs`): a core never edited per repo, plus a JSON config carrying the repo-specific inputs (roots, extensions, exclusions, map path). The **"agnostic and config-driven" requirement applies only to Phase B.** Its real targets are the **downstream Ember consumer apps** (whose code still uses the pre-carbonization tokens), driven by the map produced in Phase A. It is **not** meant to run against this monorepo's feature branch ŌĆö the carbonization refactor already updated every token consumer here ŌĆö so the only use of Phase B *inside* this repo is as a **test harness against `main`** (where the old token usages still exist) to validate the tool before pointing it at other codebases.

## 2. Branches (pre vs post)

- **Pre-carbonization** = `main` (the token set before the refactor).
- **Post-carbonization** = the Project Solar feature branch `project-solar/00_phase-1-main-feature-branch` ŌĆö this is the integration branch that accumulates the carbonization work.
- This tooling is **developed** on `project-solar/phase-2/10/HDS-6504/full-tokens-replacement`, which is branched from the feature branch and will be **merged back into it**. So when the tooling runs from this dev branch (or any branch off the feature branch), the **working tree already reflects the post-carbonization token set** ŌĆö no separate checkout is needed for "post".

Because **Phase A runs only once on this monorepo**, these refs are **fixed, not configurable**: pre = `main` (read via `git show`), post = the current **working tree** (which already reflects post-carbonization when running from the feature branch or a branch off it). Only **Phase B** is parameterized per repo ŌĆö via the token map and its own config, never via git refs.

## 3. Key facts (verified during planning)

- **`packages/tokens/dist/products/css/tokens.css` is committed** (the tokens package `.gitignore` only ignores `node_modules/` and `.DS_Store`). So the pre and post token **name** sets can be read directly from git refs ŌĆö e.g. `git show main:packages/tokens/dist/products/css/tokens.css` vs. the working tree ŌĆö **without running a build**.
- **Prefix landscape (verified by recon):** the pre file (`main`) declares **442** custom properties, **all `--token-*`** (zero `--hds-*`); the post file (working tree) declares **801**, **all `--hds-*`** (zero `--token-*`). So the two name sets live in **different prefixes**, and the `--token-` ŌåÆ `--hds-` swap is a **universal first step** that must be applied before pre/post names can be compared meaningfully. Comparing the prefix-swapped pre set against post: **242** match verbatim (mechanical-only), **200** do not (non-mechanical refactor), and **559** post names have no prefix-swapped pre counterpart (new tokens + rename targets).
- The file lists one custom property per line under `:root`. **Only the names matter** for this migration; values are ignored:

  ```css
  :root {
    --hds-border-radius-medium: 6px;
    --hds-border-color-primary: #656a7633;
    --hds-core-color-blue-400: #0046d1;
    --hds-spacing-4: 4px;
  }
  ```

- **`dist/products/css/tokens.css` is the canonical token inventory** consumers read (it is the same file the `verify-tokens` auditor uses). Themed variants under `themed-tokens/**` and `helpers/**` are **out of scope** for the canonical set.
- **Source JSON** lives under `packages/tokens/src/**` in nested DTCG format. A token's CSS name is the **prefix + the dash-joined path of object keys** down to the leaf (a leaf is an object carrying `$value`/`$type`). Verified example ŌĆö `packages/tokens/src/global/color/semantic-border.json` exists at the **same path** on both refs, but its tree was reshaped: on `main` the path `color.border.primary` ŌåÆ `--token-color-border-primary`, and on post `border.color.primary` ŌåÆ `--hds-border-color-primary` (post files also add `$modes` and switch value refs from `{color.palette.*}` to `{core.color.*}`, but those are values and thus irrelevant). Per the maintainer: only a **few** source files were renamed; most changes are **token refactors inside** the files ŌĆö so **file path is a strong provenance anchor** (see S2). The `packages/tokens/src/carbon-extracted/**` subfolder is **ignored entirely** ŌĆö it was not changed by carbonization and is not relevant to this migration.
- **Changeset files** (`.changeset/*.md`) contain **human-authored rename tables** ŌĆö the strongest available signal. Recon found **14** changesets, of which **5** carry rename tables (`cyan-boxes-add`, `silent-birds-flash`, `tall-trains-bathe`, `twelve-mirrors-grab`, `twenty-ads-travel`). Two notations are used:
  - Markdown tables: `| Before | After |` with backtick-wrapped token names.
  - Arrow lists: `` `old` ŌåÆ `new` `` (sometimes bare `hds-*`, no leading `--`).

  **Important caveats:**
  - **Changeset names are in the post `--hds-` namespace on _both_ sides** ŌĆö they document renames **after** the prefix swap has already been applied (e.g. `cyan-boxes-add` maps `--hds-color-border-primary` ŌåÆ `--hds-border-color-primary`, not the raw `--token-*` pre name). This means the prefix swap (**S0**) is the first link and changeset chains (**S1**) compose **on top of it**: `--token-color-border-primary` ŌåÆ(S0)ŌåÆ `--hds-color-border-primary` ŌåÆ(S1)ŌåÆ `--hds-border-color-primary` (Ōłł post). Verified documented refactors: `cyan-boxes-add` = `color-[semantic]` ŌåÆ `[semantic]-color`; `silent-birds-flash` = `color-palette` ŌåÆ `core-color`; `tall-trains-bathe` = `color-<product>` ŌåÆ `product-<product>-ŌĆ”-color`.
  - Some changesets are **prose placeholders** with no mappings (e.g. `big-cups-send.md`: "Updated components' code to reflect change of semantic 'product' tokens naming."). The parser must skip these.
  - A changeset documents **one step** of what may be **several successive renaming steps**. So a changeset's `before ŌåÆ after` is **not necessarily the overall pre/post delta** ŌĆö an early step's `after` may be renamed again by a later step. The tool must **chain/compose** rename pairs across changesets (and on top of the S0 prefix swap) and **reconcile the endpoints against the actual pre/post name sets** (final `before` Ōłł removed, final `after` Ōłł added). Still, changesets remain a very strong signal.
  - Changesets span the **full breadth of carbonization** (both the mechanical prefix swaps and the non-mechanical refactors), and some also reference component-internal `--hds-var-*` variables that never appear in `tokens.css`. The tool does **not** try to label which "kind" a pair is; it simply keeps the pairs whose composed endpoints match the **actual pre/post token-name delta** and records the rest as out-of-scope for this run.
- **Environment:** the reusable `verify-tokens.mjs` core (config parsing, file walking, glob matching, comment stripping, interpolation detection, MD/JSON report writers) is the architectural template to reuse for both phases.

## 4. Rename-inference signal sources (priority order)

Phase A resolves an `after` (a post name) for every pre token, then **groups the results by how each token changed** ŌĆö because the universal `--token-` ŌåÆ `--hds-` prefix swap means no name is literally unchanged, a plain pre/post name diff would be uninformative (everything "removed"). The map's mutually-exclusive top-level categories are therefore transformation-based, not identity-based: `prefix-only` (only the prefix changed), the BEM-named `prefix-plus-renaming__*` families (prefix **and** a structural rename), `removed` (pre token with no successor), and `added` (brand-new post token). Phase A resolves the `after` by combining the signals below, in priority order; each proposal records **which signals agreed** so the reviewer can audit it, and conflicts between signals are **surfaced, never hidden** (guiding principle inherited from the verify tooling). Category assignment itself is **descriptive** ŌĆö a resolved `before`ŌåÆ`after` pair is only placed in a `__palette-colors` / `__product-colors` / `__semantic-colors` family if the actual pair fits that rule, otherwise it falls to `__other` (see ┬¦5).

| # | Signal | Source | Confidence |
| --- | --- | --- | --- |
| **S0** | **Mechanical prefix swap** (`--token-` ŌåÆ `--hds-`) | rule (applies to every pre token) | High / mechanical |
| **S1** | **Changeset rename tables/arrows** (chained across steps, on top of S0) | `.changeset/*.md` | High / authoritative |
| **S2** | **Source-JSON provenance & structure** (same-file grouping + tree diff) | `packages/tokens/src/**/*.json` | High when structural |
| **S3** | **Fuzzy name similarity** | derived from the removed/added name sets | Low (review) |

- **S0 ŌĆö Prefix swap.** The baseline mechanical step applied to **every** pre token: `--token-X` ŌåÆ candidate `--hds-X`. If the candidate exists verbatim in the post set (**242** of 442 tokens), that is a **high-confidence** rename with no further inference needed. If it does not (**200** tokens), S0 still produces the correct *intermediate* name that S1/S2 chain onward from (changeset `before` values live in this `--hds-` namespace), and it is also the base for S3 fuzzy scoping. S0 is why the map's `before` values keep their raw `--token-*` form (what consumer code actually contains) while every `after` is an `--hds-*` post name.
- **S1 ŌĆö Changesets.** Parse both notations; extract every documented `old ŌåÆ new` pair; **skip placeholder changesets**. Changeset names are already in the post `--hds-` namespace, so **compose S1 pairs on top of the S0 prefix swap** and chain across steps (AŌåÆB, BŌåÆC ŌćÆ AŌåÆC), keeping only resolved endpoints whose composed `before` traces back to a pre token and whose `after` is in the post set. Pairs whose endpoints don't match the actual pre/post token-name delta (e.g. component-internal `--hds-var-*` variables that aren't in `tokens.css`) are recorded but flagged out-of-scope for this run.
- **S2 ŌĆö Source JSON.** Two complementary uses. **(a) File provenance (strong scoping signal):** the source JSON **file names were almost entirely unchanged** by carbonization (only a few files were renamed) ŌĆö so the tokens defined in a given file are **strongly correlated across pre/post**. Map each token name (pre and post) back to the source file that defines it; a `removed` token and an `added` token that originate from the **same file** are very likely a rename pair, which sharply narrows the candidate set (even when the internal DTCG tree was reshaped by the naming refactor). For the few renamed files, carry a small preŌåÆpost filename mapping so provenance still lines up. **(b) Structure/tree diff:** within a file, derive the kebab token name from each DTCG path in the pre and post trees and diff to reveal renamed groups/keys. Together these corroborate S1 and catch renames not documented in changesets.
- **S3 ŌĆö Fuzzy name similarity.** Kebab-segment-aware similarity (segment Jaccard + Levenshtein) over the removed/added sets; used only as a **fallback** for tokens left unmatched after S1ŌĆōS2. **Scoped by file provenance where possible** ŌĆö comparing a `removed` token first against the `added` tokens from the **same source file** greatly improves precision. Always emitted for human review, never auto-confirmed.

> **Deliberately NOT used: generated-CSS value matching.** Matching a removed token to an added token by identical resolved value is a **weak** signal here: many tokens share a value (false positives) and carbonization changed some values (false negatives). It is intentionally excluded.

### Systematic-pattern detection (built into the categories)

Many renames are **systematic** across a group, and rather than surface them as ad-hoc suggestions, Phase A **bakes the detected patterns into the map's categories**. Three high-volume families are recognized descriptively from the resolved `before`ŌåÆ`after` pairs and emitted as their own BEM sub-categories: `prefix-plus-renaming__palette-colors` (`color-palette-{hue}-{step}` ŌåÆ `core-color-{hue}-{step}`, 42 tokens), `prefix-plus-renaming__product-colors` (`color-{product}-ŌĆ”` ŌåÆ `product-{product}-ŌĆ”-color`, 86 tokens), and `prefix-plus-renaming__semantic-colors` (`color-{semantic}-{rest}` ŌåÆ `{semantic}-color-{rest}`, 46 tokens). Together they collapse ~174 of the ~184 structural renames into three confirmable rules; whatever does not fit a rule lands in `prefix-plus-renaming__other` (10 tokens), which ŌĆö together with `removed` (16) ŌĆö is exactly the "review by hand" set. This lets the maintainer approve a whole *rule* at once and makes any outlier pop. (During implementation, if I spot further likely-standard patterns I will ask you directly before promoting them to their own category.)

### Confidence buckets

- **high** ŌĆö the mechanical prefix swap (S0) lands verbatim on a post token, and/or documented in a changeset chain (S1), and/or a clear S2 structural match, and/or a maintainer-confirmed systematic pattern.
- **low** ŌĆö S3 fuzzy only; requires a manual decision. (Both phases only distinguish confident automated proposals from ones that need human review, so everything not `high` is simply `low`.)

## 5. Phase A ŌĆö `diff-tokens.mjs` (mapping generator)

### Inputs

Phase A is a **one-off for this monorepo**, so it takes **no config and no CLI arguments** ŌĆö all its inputs are fixed HDS paths declared as constants at the top of `diff-tokens.mjs` (there is nothing to parameterize, so a config would be pure ceremony). Inputs:

- **Pre token set** ŌĆö `git show main:packages/tokens/dist/products/css/tokens.css` (the `main` baseline).
- **Post token set** ŌĆö the **working-tree** `packages/tokens/dist/products/css/tokens.css`.
- **Changesets** ŌĆö `.changeset/*.md`.
- **Source JSON** ŌĆö `packages/tokens/src/**/*.json` (excluding `packages/tokens/src/carbon-extracted/**`, which is unchanged and irrelevant), read from `main` and the working tree.

### Processing

1. **Extract names** (values ignored) from pre and post `tokens.css` into two name sets ŌĆö pre are `--token-*`, post are `--hds-*`.
2. **Apply S0** (the `--token-` ŌåÆ `--hds-` prefix swap) to every pre name to get its intermediate `--hds-*` candidate; if it exists verbatim in post ŌåÆ mechanical high-confidence rename, else it needs S1ŌĆōS3. Post names not targeted by any pre token are `added`.
3. **Extract S1** rename chains from changesets (composed on top of S0); **extract S2** structural renames from the source-JSON diff.
4. **Propose** an `after` for each pre token by combining S0 ŌåÆ S1 ŌåÆ S2 ŌåÆ S3; attach the agreeing `signals` and a `confidence`.
5. **Classify** each resolved `before`ŌåÆ`after` pair into its mutually-exclusive category (`prefix-only`, a `prefix-plus-renaming__*` family, or `removed`) descriptively ŌĆö the pair must actually fit a family's rule or it falls to `__other`.
6. **Flag conflicts** (e.g. two different candidates for the same `before`) explicitly.

### Outputs (`reports/<codebase>/`, e.g. `reports/hds/`)

- **`token-map.generated.json`** ŌĆö the proposed map and **source-of-truth** structure: a flat object whose **mutually-exclusive top-level keys are the transformation categories** and whose every child is a uniform `{ before, after }` pair (the category is the parent array's key, so no per-entry tag is needed). This is easy to read, scan, diff, and hand-edit:

  ```jsonc
  {
    // only the --token-ŌåÆ--hds- prefix changed; structure identical (242)
    "prefix-only": [
      { "before": "--token-border-radius-medium", "after": "--hds-border-radius-medium" }
    ],
    // color-palette-{hue}-{step} ŌåÆ core-color-{hue}-{step} (42)
    "prefix-plus-renaming__palette-colors": [
      { "before": "--token-color-palette-blue-500", "after": "--hds-core-color-blue-500" }
    ],
    // color-{product}-ŌĆ” ŌåÆ product-{product}-ŌĆ”-color (86)
    "prefix-plus-renaming__product-colors": [
      { "before": "--token-color-terraform-brand", "after": "--hds-product-terraform-brand-color" }
    ],
    // color-{semantic}-{rest} ŌåÆ {semantic}-color-{rest} (46)
    "prefix-plus-renaming__semantic-colors": [
      { "before": "--token-color-border-primary", "after": "--hds-border-color-primary" }
    ],
    // structural renames with no systematic rule ŌĆö review each (10)
    "prefix-plus-renaming__other": [
      { "before": "--token-focus-ring-action-box-shadow", "after": "--hds-focus-ring-box-shadow-action" }
    ],
    // pre token with no successor ŌĆö decide manually or flag with a TODO (16)
    "removed": [
      { "before": "--token-app-header-logo-size", "after": null }
    ],
    // brand-new post tokens (before:null) ŌĆö reference only; Phase B ignores these (377)
    "added": [
      { "before": null, "after": "--hds-brand-new-token" }
    ]
  }
  ```

  Every pre token lands in exactly one of `prefix-only`, `prefix-plus-renaming__*`, or `removed`, so the map doubles as a complete inventory of what happened to each old token. Phase B consumes every category uniformly ŌĆö a non-null `after` is a rename, `after: null` is a TODO drop, and `before: null` (the `added` category) is ignored ŌĆö so the key names are purely for human legibility and never affect behavior.
- **`token-diff.md`** ŌĆö the detailed human report: per-category and per-signal summary counts, one section per category (with each family's rule stated) carrying a `signals` column for confidence, and any changeset conflicts. This is where a reviewer goes for depth; the JSON stays minimal and editable.

### Confirming the map

The human reviews `token-diff.md`, edits/completes `token-map.generated.json`, and saves it as **`token-map.json`** ŌĆö the confirmed source of truth consumed by Phase B. Phase A always (re)writes `token-map.generated.json` (safe to overwrite on every run) and never overwrites the human-owned `token-map.json`.


## 6. Phase B ŌĆö `migrate-tokens.mjs` (applier) + verification

> **Depends on the human-confirmed `token-map.json`.** Phase B **never** applies the raw generated proposal ŌĆö only the reviewed, confirmed map.

### Inputs

Again, the only CLI argument is `--config <path>`. The config points at the confirmed `token-map.json` and the scan roots/extensions/excludes (HDS defaults: `packages/components/src`, `showcase/app`, `showcase/tests`, `website/app`, `website/docs`; extensions `scss, css, gts, hbs, ts, js`). For the HDS test-harness the scan `prefix` is **`--token-`** (the pre prefix that still exists in `main`), since that is what every `before` name carries.

### Processing

1. Load the confirmed map; iterate **every category array uniformly** and build a `before ŌåÆ after` lookup from entries with a non-null `after` (a string `before` and string `after`). Entries with `after: null` become TODO drops; entries with `before: null` (the `added` category) are ignored outright ŌĆö nothing to rewrite there. Because consumption is category-agnostic, adding or renaming sub-categories in the map never requires touching Phase B.
2. Walk configured roots (reusing the verify core's `collectFiles` / `walk` / glob excluder / `stripComments`).
3. Replace `var(--old)` and bare `--old` custom-property references with `--new`. **Mechanical, idempotent, position-preserving.** (For HDS, `--old` is `--token-*` and `--new` is `--hds-*`.)
4. **Skip interpolated** dynamic names (`--token-ŌĆ”#{ŌĆ”}` / `--token-ŌĆ”${ŌĆ”}`) and **flag** them for manual review (cannot be statically rewritten).
5. **Removed tokens with no confirmed `after`** (`after: null`) ŌåÆ insert a `­¤Ü¦ TODO [HDS-TOKEN-CARBONIZATION]` marker comment at the usage site instead of editing (mirrors the convention in `copilot-plans/hds-tokens-replacement/`), and report each occurrence.
6. When `dryRun: true` in config, print planned edits + counts and write no source files (report is still written).

### Outputs

Written to `reportDir` (default `reports/<codebase>/` ŌĆö the same folder as the map).

- **`token-migration.md`** ŌĆö the **human-readable** report: replacements per token, files changed, skipped interpolations, and unmapped-token TODO locations. This is what a reviewer reads.
- **`token-migration.json`** ŌĆö the **machine-readable** twin of the same data (structured, stable shape), for programmatic use: CI assertions (e.g. ŌĆ£fail if any `after:null` usages remainŌĆØ), diffing runs against each other, driving follow-up scripts, and keeping an audit trail. It is **not** consumed by Phase B itself ŌĆö it's an output artifact for downstream tooling/automation.

### Verification pass

Re-scan the roots (reusing the `verify-tokens` scan) and confirm **zero stale pre-only token names** remain (excluding intentionally-flagged / interpolated cases). Report any residuals.

Phase A (`diff-tokens.mjs`) is monorepo-specific and is **not** carried to other repos.

## 8. Decisions & default choices

1. **Names only, values ignored** ŌĆö the migration rewrites token names; token values (even where they changed during carbonization) are irrelevant.
2. **No value-match signal** ŌĆö intentionally excluded as too weak (false positives/negatives); see ┬¦4.
3. **Config-only CLI (Phase B)** ŌĆö Phase B's sole CLI argument is `--config`; all its inputs live in the config file. Phase A takes **no config and no arguments** (fixed one-off paths hardcoded in the script).
4. **Phase A is one-off; Phase B is portable** ŌĆö Phase A runs once here with fixed HDS inputs (pre `main`, post working tree); only Phase B and the token map travel to other repos.
5. **Source-of-truth token map** ŌĆö a flat, hand-editable JSON object whose mutually-exclusive keys are the transformation categories (`prefix-only`, `prefix-plus-renaming__*`, `removed`, `added`) and whose children are uniform `{ before, after }` pairs; it doubles as tooling input, human reference, and documentation source. Phase B consumes every category uniformly (non-null `after` ŌåÆ rename, `after: null` ŌåÆ TODO, `before: null` ŌåÆ ignored), so category names are for legibility only.
6. **Sass `$hds-*` / `$token-*` variables** ŌĆö CSS custom properties only by default; a configurable `sassPrefixes` option enables Sass-variable coverage per repo when needed.
7. **Two-phase separation** ŌĆö Phase B consumes **only** the human-confirmed `token-map.json`; the generated proposal is never applied automatically.
8. **Scope** ŌĆö `--hds-*` token usages in source; excludes `dist/**`, `node_modules`, `*.map`, `*.md` (as edit targets), compiled showcase assets, and `themed-tokens/**`.
9. **Config paths are resolved relative to the config file** ŌĆö `mapPath` and `reportDir` are joined onto the config's own directory, not the scanned repo root (so a `--root` pointing elsewhere still finds the map). Because the config lives in `tooling/config/` while reports live in `tooling/reports/`, the HDS config uses `../reports/hds/...` values. (Learned during implementation: an initial `reports/hds/...` value resolved to `tooling/config/reports/hds/...` and failed.)
10. **TODO-marker insertion must be idempotent** ŌĆö inserting an `after: null` marker checks whether the preceding line already carries the marker and skips if so; otherwise re-runs stack duplicate comments (caught in idempotency testing).

## 9. Verification strategy

1. **Phase A sanity check** ŌĆö run the diff (pre `main` vs post working tree); open `token-diff.md`; confirm changeset chains resolve to real pre/post endpoints; spot-check a few renames against the source JSON under `packages/tokens/src/products`; confirm the systematic-pattern families. **Actual first run:** 442 pre ŌåÆ 801 post, categorized as **prefix-only 242**, **prefix-plus-renaming__palette-colors 42**, **__product-colors 86**, **__semantic-colors 46**, **__other 10**, **removed 16**, plus **377** brand-new post tokens (`added`); by signal: **419 high-confidence** (242 S0 prefix-swap, 176 S1 changeset, 1 S2 structural), **7 S3 fuzzy**, **16 unresolved** (`after: null`), 0 changeset conflicts. The by-hand review set is `__other` (10) + `removed` (16).
2. **Phase B dry-run** ŌĆö set `dryRun: true`; inspect planned edits and counts before touching files.
3. **Test harness against `main`** ŌĆö the working tree here is already post-carbonization (consumers use `--hds-*`, only excluded `.md` docs still reference `--token-*`), so validate Phase B against real pre-token usage via a throwaway worktree: `git worktree add /tmp/hds-main main`, run Phase B with `--root /tmp/hds-main`, then `git worktree remove /tmp/hds-main --force`. **Actual validated run:** 2970 files scanned, 99 changed, **1176 replacements**, 34 TODO markers, 33 interpolated skipped, **0 stale remaining**; `.gts`/`.hbs`/`.scss` all rewritten correctly (e.g. `var(--token-color-terraform-brand)` ŌåÆ `var(--hds-product-terraform-brand-color)`).
4. **Apply on a clean checkout** ŌĆö review the Git diff; run `pnpm lint` / `pnpm lint:fix`. **Do not run the test suite from the agent** ŌĆö ask the maintainer to run tests and report back.
5. **Idempotency** ŌĆö re-running from a clean checkout must produce a **byte-for-byte identical diff**. **Validated:** a second apply reported `changed=0, replacements=0, stale-remaining=0` (after fixing non-idempotent TODO-marker insertion; see ┬¦8.10).
6. **Post-apply verify pass** ŌĆö reports **0 stale pre-token names** (excluding intentionally flagged / interpolated usages).

## 10. Open questions / risks

- **Multi-step rename chains** ŌĆö the biggest inference subtlety: a changeset's `before ŌåÆ after` may be an intermediate step. Endpoints must be composed and reconciled against the actual pre/post name sets, or the map will point at a name that no longer exists.
- **Changeset coverage vs. completeness** ŌĆö if some carbonization renames are only in placeholder changesets, S2ŌĆōS3 must fill the gap; those land in low confidence and need manual confirmation. The `token-diff.md` unmatched tables make them visible.
- **Broad changeset scope** ŌĆö changesets cover the whole carbonization effort (mechanical prefix swaps and non-mechanical refactors) and sometimes reference component-internal `--hds-var-*` variables absent from `tokens.css`; scoping composed endpoints against the actual pre/post name delta filters out anything that isn't part of this run's token migration, but review is still advised.
- **Systematic patterns need confirmation** ŌĆö bulk pattern application is powerful but must be maintainer-confirmed to avoid over-applying a rule to tokens that were actually handled differently.
- **Indirect (non-`--`) coupling** ŌĆö code that couples to a bare token `name` string (JS/TS reading the tokens JSON) won't be caught by CSS-usage scanning; these are flagged with the TODO marker for manual review, consistent with the replacement plan.
