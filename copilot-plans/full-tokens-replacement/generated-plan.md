# Generated Plan — HDS Token "Carbonization" Migration Tooling

> Two-phase, config-driven, dependency-free tooling to migrate CSS design-token
> usages from the **pre-carbonization** token set (`main`) to the
> **post-carbonization** token set (branch
> `project-solar/phase-2/10/HDS-6504/full-tokens-replacement`).
>
> **Phase A** generates a *proposed* old→new token mapping for human review and
> confirmation. **Phase B** applies the *confirmed* mapping to consumer code and
> verifies the result.
>
> Status: **PLAN — not yet executed.** Awaiting green light.
> Last updated: 2026-07-15

---

## 1. Context & goal

The Helios Design System (HDS) design tokens are generated by `packages/tokens`
(a style-dictionary build) and consumed as CSS custom properties throughout the
component library, the showcase app, the website, and downstream consumer apps.

As part of **Project Solar**, the token set is undergoing a large **"carbonization"**
refactor. Unlike the earlier mechanical efforts:

- **`--token-*` → `--hds-*`** — a pure prefix swap (see
  `copilot-plans/hds-tokens-replacement/`).
- **`--hds-*` → `--hds-var-*`** — component-internal variable renaming (see
  `copilot-plans/hds-var-renaming/`).

…carbonization is a **non-mechanical** refactor: tokens are **renamed, removed,
added, and re-valued** in arbitrary ways (e.g. the semantic color namespace moved
from `color-[semantic]` to `[semantic]-color`:
`--hds-color-border-primary` → `--hds-border-color-primary`). There is **no simple
find-and-replace rule** — the migration requires an explicit old→new mapping.

### Goal

Produce reusable tooling that:

1. **Phase A — mapping generation:** diffs the pre- and post-carbonization token
   sets and produces a **proposed old→new rename map**, drawing on multiple signal
   sources (authoritative changeset tables, source-JSON structure, generated-CSS
   value matching, and fuzzy name similarity), ranked by confidence, for **human
   review and confirmation**.
2. **Phase B — migration:** takes the **human-confirmed** map and mechanically
   rewrites every stale token usage in consumer code to the new name, flags
   anything with no mapping for manual review, and verifies zero stale token names
   remain.

The tooling must stay **agnostic and config-driven** (like
`copilot-plans/hds-tokens-verify/tooling/verify-tokens.mjs`): a reusable core that
is never edited per repo, plus a JSON config carrying all repo-specific inputs, so
it can later be pointed at downstream Ember consumer apps.

## 2. Key facts (verified during planning)

- **`packages/tokens/dist/products/css/tokens.css` is committed** (the tokens
  package `.gitignore` only ignores `node_modules/` and `.DS_Store`). So the pre
  and post token sets can be read directly from git refs — e.g.
  `git show main:packages/tokens/dist/products/css/tokens.css` vs. the branch —
  **without running a build**.
- The file lists resolved values, one per line, under `:root`:

  ```css
  :root {
    --hds-border-radius-medium: 6px;
    --hds-border-color-primary: #656a7633;
    --hds-core-color-blue-400: #0046d1;
    --hds-spacing-4: 4px;
  }
  ```

  Because **values are present**, exact value matching is available as a strong
  corroborating signal for renames.
- **`dist/products/css/tokens.css` is the canonical token inventory** consumers
  read (it is the same file the `verify-tokens` auditor uses). Themed variants
  under `themed-tokens/**` and `helpers/**` are **out of scope** for the canonical
  set.
- **Source JSON** lives under `packages/tokens/src/**` in nested DTCG format
  (e.g. `tag.border.radius` → `--hds-tag-border-radius`). Per the maintainer:
  only a **few** source files were renamed; most changes are **token refactors
  inside** the files.
- **Changeset files** (`.changeset/*.md`) contain **authoritative, human-authored
  rename tables** — the strongest signal. Two notations are used:
  - Markdown tables: `| Before | After |` with backtick-wrapped token names.
  - Arrow lists: `` `old` → `new` `` (sometimes bare `hds-*`, no leading `--`).

  **Caveat:** some changesets are **prose placeholders** with no mappings (e.g.
  `big-cups-send.md`: "Updated components' code to reflect change of semantic
  'product' tokens naming."). The parser must skip these. Changesets also **mix
  multiple efforts** (carbonization renames, `--token`→`--hds`, `--hds`→`--hds-var`),
  so extracted pairs must be scoped/classified against the actual token diff.
- **Environment:** the reusable `verify-tokens.mjs` core (config parsing, file
  walking, glob matching, comment stripping, interpolation detection, MD/JSON
  report writers) is the architectural template to reuse for both phases.

## 3. Rename-inference signal sources (priority order)

Phase A combines four signals. Each proposed rename records **which signals agreed**
so the human reviewer can audit it. Conflicts between signals are **surfaced, never
hidden** (guiding principle inherited from the verify tooling).

| # | Signal | Source | Confidence |
| --- | --- | --- | --- |
| **S1** | **Changeset rename tables/arrows** | `.changeset/*.md` | **Authoritative / high** |
| **S2** | **Source-JSON before/after structure** | `packages/tokens/src/**/*.json` | High when structural |
| **S3** | **Generated-CSS value match** | `dist/products/css/tokens.css` (pre vs post) | Corroborating |
| **S4** | **Fuzzy name similarity** | derived from the removed/added name sets | Medium / low (review) |

- **S1 — Changesets (authoritative).** Parse both notations; extract every
  documented `old → new` pair. Skip placeholder changesets. Because changesets span
  multiple efforts, only pairs whose `old` name is in the *removed* set and `new`
  name is in the *added* set (per the S3 CSS diff) are treated as in-scope
  carbonization renames; others are recorded but flagged out-of-scope.
- **S2 — Source JSON.** Derive the kebab token name from each DTCG path in the pre
  and post trees; diff to reveal renamed groups/keys. Useful to corroborate S1 and
  to catch renames not documented in changesets.
- **S3 — Generated CSS value match.** For each removed token, an added token with an
  **identical resolved value** is a strong rename candidate. Value match alone is
  not conclusive (many tokens share a value), so it **raises confidence** on top of
  S1/S2/S4 rather than standing alone.
- **S4 — Fuzzy name similarity.** Kebab-segment-aware similarity (segment Jaccard +
  Levenshtein) over the removed/added sets; used only as a **fallback** for tokens
  left unmatched after S1–S3. Always emitted for human review, never auto-confirmed.

**Confidence buckets:**

- **authoritative** — documented in a changeset (S1), optionally corroborated by
  S2/S3.
- **high** — S2 structural match and/or S3 value match with a close name.
- **medium / low** — S4 fuzzy only; requires manual decision.

## 4. Phase A — `diff-tokens.mjs` (mapping generator)

### Inputs

- **Pre token set** — default `git show <preRef>:<tokensCssPath>` (`preRef` default
  `main`); overridable with `--pre-css <path>`.
- **Post token set** — default: the working-tree
  `packages/tokens/dist/products/css/tokens.css`; overridable with
  `--post-ref <ref>` / `--post-css <path>`.
- **Changesets** — `.changeset/*.md` (configurable dir).
- **Source JSON** — `packages/tokens/src/**/*.json` from pre ref and post
  (working tree / ref), for S2.

### Processing

1. **Parse** pre and post `tokens.css` into `name → value` maps (regex or PostCSS,
   mirroring `loadOfficialTokens` in the verify core).
2. **Classify** every token:
   - **unchanged** — present in both, same value.
   - **value-changed** — present in both, **same name**, different value → **no
     consumer migration needed** (the usage still resolves); recorded as a note.
   - **removed** — present only in pre (migration candidates).
   - **added** — present only in post (rename targets).
3. **Extract S1** rename pairs from changesets; **extract S2** structural renames
   from the JSON diff.
4. **Match** each removed token to an added token by combining S1–S4; attach
   `confidence` and `signals[]` (which sources agreed).
5. **Detect conflicts** (e.g. a changeset says A→B but value match says A→C) and
   flag them explicitly.

### Outputs (`reports/<subdir>/`)

- **`token-diff.json`** — full structured data:

  ```jsonc
  {
    "generatedAt": "…",
    "preRef": "main",
    "postRef": "working-tree",
    "summary": { "removed": 0, "added": 0, "valueChanged": 0, "renames": 0, "unmatchedRemoved": 0, "unmatchedAdded": 0 },
    "renames": [
      { "from": "--hds-color-border-primary", "to": "--hds-border-color-primary", "confidence": "authoritative", "signals": ["changeset", "valueMatch"] }
    ],
    "valueChanged": [ { "token": "…", "before": "…", "after": "…" } ],
    "removedUnmatched": [ "…" ],
    "addedUnmatched": [ "…" ],
    "conflicts": [ { "from": "…", "candidates": ["…"], "reason": "…" } ]
  }
  ```

- **`token-diff.md`** — human-readable review report: summary table, renames grouped
  by confidence (authoritative → high → medium → low), a clearly separated
  "value-changed (no action)" section, and "unmatched removed / added" tables for
  manual attention. Changeset-sourced vs. inferred renames are visually
  distinguished.
- **`token-rename-map.template.json`** — a `{ "from": "to" }` object **pre-filled
  with authoritative + high-confidence** renames and **commented blanks** for the
  rest. The human edits/approves this into the **confirmed map** consumed by
  Phase B. Nothing below high confidence is pre-filled.

### CLI (indicative)

```bash
node copilot-plans/full-tokens-replacement/tooling/diff-tokens.mjs \
  --config copilot-plans/full-tokens-replacement/tooling/config/hds.config.json \
  --pre-ref main \
  --post-ref project-solar/phase-2/10/HDS-6504/full-tokens-replacement
```

## 5. Phase B — `migrate-tokens.mjs` (applier) + verification

> **Depends on the human-confirmed rename map.** Phase B **never** applies raw
> proposals — only the reviewed, approved map.

### Inputs

- **Confirmed rename map** — `{ "--hds-old": "--hds-new", … }` (the approved
  descendant of `token-rename-map.template.json`).
- **Scan roots / extensions / excludes** — from config (HDS defaults:
  `packages/components/src`, `showcase/app`, `showcase/tests`, `website/app`,
  `website/docs`; extensions `scss, css, gts, hbs, ts, js`).

### Processing

1. Walk configured roots (reusing the verify core's `collectFiles` / `walk` / glob
   excluder / `stripComments`).
2. Replace `var(--old)` and bare `--old` custom-property references with `--new`.
   **Mechanical, idempotent, position-preserving.**
3. **Skip interpolated** dynamic names (`--hds-…#{…}` / `--hds-…${…}`) and **flag**
   them for manual review (cannot be statically rewritten).
4. **Removed tokens with no confirmed mapping** → insert a
   `TODO [HDS-TOKEN-CARBONIZATION]` marker comment at the usage site instead of
   editing (mirrors the convention in `copilot-plans/hds-tokens-replacement/`), and
   report each occurrence.
5. Support **`--dry-run`** (print planned edits + counts, write no files).

### Outputs

- **`token-migration.md` / `.json`** — replacements per token, files changed, skipped
  interpolations, and unmapped-token TODO locations.

### Verification pass

Re-scan the roots (reusing the `verify-tokens` scan) and confirm **zero stale
pre-only token names** remain (excluding intentionally-flagged/interpolated cases).
Report any residuals.

### CLI (indicative)

```bash
# Dry run first
node copilot-plans/full-tokens-replacement/tooling/migrate-tokens.mjs \
  --config copilot-plans/full-tokens-replacement/tooling/config/hds.config.json \
  --map copilot-plans/full-tokens-replacement/tooling/reports/hds/token-rename-map.confirmed.json \
  --dry-run

# Apply
node copilot-plans/full-tokens-replacement/tooling/migrate-tokens.mjs \
  --config … --map …
```

## 6. Proposed file structure

```
copilot-plans/full-tokens-replacement/
├── generated-plan.md                     # this file
└── tooling/
    ├── diff-tokens.mjs                    # Phase A — mapping generator (reusable core)
    ├── migrate-tokens.mjs                 # Phase B — applier + verification (reusable core)
    ├── README.md                          # usage, options, workflow
    ├── config/
    │   └── hds.config.json                # repo-specific config for THIS monorepo
    └── reports/
        └── hds/
            ├── token-diff.md              # Phase A human report
            ├── token-diff.json            # Phase A machine data
            ├── token-rename-map.template.json   # generated proposal → human edits this
            ├── token-rename-map.confirmed.json  # human-approved input to Phase B
            ├── token-migration.md         # Phase B report
            └── token-migration.json
```

The two `.mjs` scripts are the **reusable core** (never edited per repo). Retargeting
to a downstream consumer app = copy the scripts unchanged + author a new
`config/<repo>.config.json`.

### Config reference (`config/hds.config.json`)

| Key | Purpose | HDS default |
| --- | --- | --- |
| `tokensCssPath` | Path to canonical `tokens.css` within the tokens package/repo | `packages/tokens/dist/products/css/tokens.css` |
| `preRef` | Git ref for the pre-carbonization token set | `main` |
| `postRef` | Git ref for the post set (`null` = working tree) | `null` |
| `changesetsDir` | Directory of changeset files (S1) | `.changeset` |
| `sourceJsonGlob` | Glob for DTCG source JSON (S2) | `packages/tokens/src/**/*.json` |
| `prefix` | Token prefix to migrate | `--hds-` |
| `sassPrefixes` | Optional secondary prefixes (e.g. `$hds-`) — off by default | `[]` |
| `roots` | Dirs to scan/rewrite | components/showcase/website roots |
| `extensions` | File extensions | `scss, css, gts, hbs, ts, js` |
| `excludeGlobs` | Globs to skip | `dist/**`, `node_modules/**`, `*.map`, `*.md`, compiled assets |
| `reportSubdir` | Subfolder under `reports/` | `hds` |
| `todoMarker` | Marker for unmapped removed tokens | `TODO [HDS-TOKEN-CARBONIZATION]` |

## 7. Decisions & default choices

1. **Value-match (S3)** — **included** as a corroborating signal that raises
   confidence; never the sole criterion for an auto-filled rename.
2. **Sass `$hds-*` / `$token-*` variables** — **CSS custom properties only by
   default**; a configurable `sassPrefixes` option enables Sass-variable coverage
   per repo when needed.
3. **dist freshness** — **trust the committed `tokens.css`** on each ref by default
   (dist is committed). An optional freshness check / `--build` escape hatch can be
   added if a branch's `dist` is found stale.
4. **Two-phase separation** — Phase B consumes **only** the human-confirmed map;
   proposals are never applied automatically.
5. **Scope** — `--hds-*` token usages in source; excludes `dist/**`, `node_modules`,
   `*.map`, `*.md` (as edit targets), compiled showcase assets, and
   `themed-tokens/**`.

## 8. Verification strategy

1. **Phase A sanity check** — run the diff on `main` vs the post branch; open
   `token-diff.md`; confirm S1 changeset pairs and S3 value matches agree; spot-check
   a few renames against the source JSON under `packages/tokens/src/products`.
2. **Phase B dry-run** — run with `--dry-run`; inspect planned edits and counts
   before touching files.
3. **Apply on a clean checkout** — review the Git diff; run `pnpm lint` /
   `pnpm lint:fix`. **Do not run the test suite from the agent** — ask the maintainer
   to run tests and report back.
4. **Idempotency** — re-running from a clean checkout must produce a **byte-for-byte
   identical diff**.
5. **Post-apply verify pass** — reports **0 stale pre-token names** (excluding
   intentionally flagged / interpolated usages).

## 9. Open questions / risks

- **Changeset coverage vs. completeness** — if some carbonization renames are not
  documented in any changeset (only placeholders), S2–S4 must fill the gap; those
  land in medium/low confidence and need manual confirmation. The `token-diff.md`
  "unmatched removed / added" tables make these visible.
- **Multi-effort changeset noise** — changesets mix carbonization with the
  `--token`→`--hds` and `--hds`→`--hds-var` efforts; scoping against the actual
  pre/post CSS diff filters out-of-scope pairs, but review is still advised.
- **Value collisions** — many tokens share a value (e.g. alpha colors), so S3 must
  only corroborate, not decide.
- **Stale `dist` on a branch** — if the post branch's `dist` wasn't rebuilt after a
  source change, the token set could be inaccurate; see decision §7.3.
- **Indirect (non-`--`) coupling** — code that couples to a bare token `name` string
  (JS/TS reading the tokens JSON) won't be caught by CSS-usage scanning; these are
  flagged with the TODO marker for manual review, consistent with the replacement
  plan.

## 10. Execution order (once approved)

1. Scaffold `tooling/` (scripts, `config/hds.config.json`, `README.md`).
2. Implement **Phase A** (`diff-tokens.mjs`); run against `main` vs the post branch;
   iterate on `token-diff.md` until the proposed map is trustworthy.
3. Human reviews `token-diff.md` and produces `token-rename-map.confirmed.json`.
4. Implement **Phase B** (`migrate-tokens.mjs`); `--dry-run`; review; apply on a
   clean checkout; `pnpm lint`; maintainer runs tests.
5. Fold every learning back into this plan and the tooling `README.md` (the plan is
   the durable, portable artifact — mirroring the convention of the sibling plans).
