# Generated Plan — HDS Token "Carbonization" Migration Tooling

> Two-phase, config-driven, dependency-free tooling to migrate CSS design-token usages from the **pre-carbonization** token set to the **post-carbonization** token set.
>
> **Phase A** generates a proposed old→new token map (an editable JSON that becomes the shared source of truth) for human review. **Phase B** applies the confirmed map to consumer code and verifies the result.
>
> Status: **PLAN — not yet executed.** Awaiting green light.
> Last updated: 2026-07-16

---

## 1. Context & goal

The Helios Design System (HDS) design tokens are generated by `packages/tokens` (a style-dictionary build) and consumed as CSS custom properties throughout the component library, the showcase app, the website, and downstream consumer apps.

As part of **Project Solar**, the token set is undergoing a large **"carbonization"** refactor. Carbonization is an **umbrella effort** that bundles together several kinds of token change accumulated across the work — both **mechanical** and **non-mechanical**:

- **Mechanical renames**, e.g. the `--token-*` → `--hds-*` prefix swap (see `copilot-plans/hds-tokens-replacement/`) and the `--hds-*` → `--hds-var-*` component-internal variable renaming (see `copilot-plans/hds-var-renaming/`) — simple, rule-based transformations.
- **Non-mechanical refactors** — tokens **renamed, removed, and added** in arbitrary ways (e.g. the semantic color namespace moved from `color-[semantic]` to `[semantic]-color`: `--hds-color-border-primary` → `--hds-border-color-primary`).

Because the non-mechanical part has **no single find-and-replace rule**, the overall migration requires an explicit old→new mapping. This tooling therefore treats carbonization as **one combined delta**: it diffs the pre- vs post-carbonization token **names** and captures whatever changed — mechanical and non-mechanical alike — in a single map. (Token *values* also changed in places, but values are irrelevant to this migration, which only rewrites token **names**.)

### Goal

Produce reusable tooling that:

1. **Phase A — mapping generation:** diffs the pre- and post-carbonization token **name** sets and produces a proposed old→new **token map** (see §4). The map is an editable JSON that serves triple duty: input to Phase B, human reference, and a basis for documentation/migration-guide generation.
2. **Phase B — migration:** takes the **human-confirmed** token map and mechanically rewrites every stale token usage in consumer code to the new name, flags anything with no mapping for manual review, and verifies zero stale token names remain.

The two phases have very different reuse profiles:

- **Phase A runs once, on this monorepo** — it is the only place with all the inputs needed to build the map (the `main` baseline, the changesets, the token source). It does **not** need to be portable. Running it here is what produces the two artifacts that then travel onward: (1) the confirmed **token map JSON**, and (2) a validated **Phase B** migration tool hardened against this repo before it is pointed at other codebases.
- **Phase B is the reusable, config-driven tool** (like `copilot-plans/hds-tokens-verify/tooling/verify-tokens.mjs`): a core never edited per repo, plus a JSON config carrying the repo-specific inputs (roots, extensions, exclusions, map path). The **"agnostic and config-driven" requirement applies only to Phase B.** Its real targets are the **downstream Ember consumer apps** (whose code still uses the pre-carbonization tokens), driven by the map produced in Phase A. It is **not** meant to run against this monorepo's feature branch — the carbonization refactor already updated every token consumer here — so the only use of Phase B *inside* this repo is as a **test harness against `main`** (where the old token usages still exist) to validate the tool before pointing it at other codebases.

## 2. Branches (pre vs post)

- **Pre-carbonization** = `main` (the token set before the refactor).
- **Post-carbonization** = the Project Solar feature branch `project-solar/00_phase-1-main-feature-branch` — this is the integration branch that accumulates the carbonization work.
- This tooling is **developed** on `project-solar/phase-2/10/HDS-6504/full-tokens-replacement`, which is branched from the feature branch and will be **merged back into it**. So when the tooling runs from this dev branch (or any branch off the feature branch), the **working tree already reflects the post-carbonization token set** — no separate checkout is needed for "post".

Because **Phase A runs only once on this monorepo**, these refs are **fixed, not configurable**: pre = `main` (read via `git show`), post = the current **working tree** (which already reflects post-carbonization when running from the feature branch or a branch off it). Only **Phase B** is parameterized per repo — via the token map and its own config, never via git refs.

## 3. Key facts (verified during planning)

- **`packages/tokens/dist/products/css/tokens.css` is committed** (the tokens package `.gitignore` only ignores `node_modules/` and `.DS_Store`). So the pre and post token **name** sets can be read directly from git refs — e.g. `git show main:packages/tokens/dist/products/css/tokens.css` vs. the working tree — **without running a build**.
- The file lists one custom property per line under `:root`. **Only the names matter** for this migration; values are ignored:

  ```css
  :root {
    --hds-border-radius-medium: 6px;
    --hds-border-color-primary: #656a7633;
    --hds-core-color-blue-400: #0046d1;
    --hds-spacing-4: 4px;
  }
  ```

- **`dist/products/css/tokens.css` is the canonical token inventory** consumers read (it is the same file the `verify-tokens` auditor uses). Themed variants under `themed-tokens/**` and `helpers/**` are **out of scope** for the canonical set.
- **Source JSON** lives under `packages/tokens/src/**` in nested DTCG format (e.g. `tag.border.radius` → `--hds-tag-border-radius`). Per the maintainer: only a **few** source files were renamed; most changes are **token refactors inside** the files. The `packages/tokens/src/carbon-extracted/**` subfolder is **ignored entirely** — it was not changed by carbonization and is not relevant to this migration.
- **Changeset files** (`.changeset/*.md`) contain **human-authored rename tables** — the strongest available signal. Two notations are used:
  - Markdown tables: `| Before | After |` with backtick-wrapped token names.
  - Arrow lists: `` `old` → `new` `` (sometimes bare `hds-*`, no leading `--`).

  **Important caveats:**
  - Some changesets are **prose placeholders** with no mappings (e.g. `big-cups-send.md`: "Updated components' code to reflect change of semantic 'product' tokens naming."). The parser must skip these.
  - A changeset documents **one step** of what may be **several successive renaming steps**. So a changeset's `before → after` is **not necessarily the overall pre/post delta** — an early step's `after` may be renamed again by a later step. The tool must **chain/compose** rename pairs across changesets and **reconcile the endpoints against the actual pre/post name sets** (final `before` ∈ removed, final `after` ∈ added). Still, changesets remain a very strong signal.
  - Changesets span the **full breadth of carbonization** (both the mechanical prefix swaps and the non-mechanical refactors), and some also reference component-internal `--hds-var-*` variables that never appear in `tokens.css`. The tool does **not** try to label which "kind" a pair is; it simply keeps the pairs whose composed endpoints match the **actual pre/post token-name delta** and records the rest as out-of-scope for this run.
- **Environment:** the reusable `verify-tokens.mjs` core (config parsing, file walking, glob matching, comment stripping, interpolation detection, MD/JSON report writers) is the architectural template to reuse for both phases.

## 4. Rename-inference signal sources (priority order)

Phase A classifies tokens by **name** into `unchanged` (name present in both pre and post), `removed` (pre only), and `added` (post only). It then proposes an `after` name for each `removed` token by combining the signals below. Each proposal records **which signals agreed** so the human reviewer can audit it, and conflicts between signals are **surfaced, never hidden** (guiding principle inherited from the verify tooling).

| # | Signal | Source | Confidence |
| --- | --- | --- | --- |
| **S1** | **Changeset rename tables/arrows** (chained across steps) | `.changeset/*.md` | High / authoritative |
| **S2** | **Source-JSON provenance & structure** (same-file grouping + tree diff) | `packages/tokens/src/**/*.json` | High when structural |
| **S3** | **Fuzzy name similarity** | derived from the removed/added name sets | Low (review) |

- **S1 — Changesets.** Parse both notations; extract every documented `old → new` pair; **skip placeholder changesets**. Because renames may span multiple steps, **compose the pairs into chains** (A→B, B→C ⇒ A→C) and keep only the resolved endpoints whose `before` is in the removed set and `after` is in the added set. Pairs whose endpoints don't match the actual pre/post token-name delta (e.g. already-in-`main` prefix swaps, or component-internal `--hds-var-*` variables that aren't in `tokens.css`) are recorded but flagged out-of-scope for this run.
- **S2 — Source JSON.** Two complementary uses. **(a) File provenance (strong scoping signal):** the source JSON **file names were almost entirely unchanged** by carbonization (only a few files were renamed) — so the tokens defined in a given file are **strongly correlated across pre/post**. Map each token name (pre and post) back to the source file that defines it; a `removed` token and an `added` token that originate from the **same file** are very likely a rename pair, which sharply narrows the candidate set (even when the internal DTCG tree was reshaped by the naming refactor). For the few renamed files, carry a small pre→post filename mapping so provenance still lines up. **(b) Structure/tree diff:** within a file, derive the kebab token name from each DTCG path in the pre and post trees and diff to reveal renamed groups/keys. Together these corroborate S1 and catch renames not documented in changesets.
- **S3 — Fuzzy name similarity.** Kebab-segment-aware similarity (segment Jaccard + Levenshtein) over the removed/added sets; used only as a **fallback** for tokens left unmatched after S1–S2. **Scoped by file provenance where possible** — comparing a `removed` token first against the `added` tokens from the **same source file** greatly improves precision. Always emitted for human review, never auto-confirmed.

> **Deliberately NOT used: generated-CSS value matching.** Matching a removed token to an added token by identical resolved value is a **weak** signal here: many tokens share a value (false positives) and carbonization changed some values (false negatives). It is intentionally excluded.

### Systematic-pattern detection (assist for the maintainer)

Many renames are **systematic** across a group (e.g. every `--hds-color-<semantic>` became `--hds-<semantic>-color`). When Phase A detects that a set of `removed` tokens share a common structural transformation that lands them on existing `added` tokens, it will **surface the candidate pattern** (the rule + the tokens it would cover) in `token-diff.md` and **pause to ask the maintainer to confirm** before treating the whole group as high-confidence. Confirmed patterns can then resolve many tokens at once, which greatly speeds up review. (During implementation, if I spot such likely-standard patterns I will ask you directly to confirm them.)

### Confidence buckets

- **high** — documented in a changeset chain (S1) and/or a clear S2 structural match, and/or a maintainer-confirmed systematic pattern.
- **low** — S3 fuzzy only; requires a manual decision. (Both phases only distinguish confident automated proposals from ones that need human review, so everything not `high` is simply `low`.)

## 5. Phase A — `diff-tokens.mjs` (mapping generator)

### Inputs

Phase A is a **one-off for this monorepo**, so it takes **no config and no CLI arguments** — all its inputs are fixed HDS paths declared as constants at the top of `diff-tokens.mjs` (there is nothing to parameterize, so a config would be pure ceremony). Inputs:

- **Pre token set** — `git show main:packages/tokens/dist/products/css/tokens.css` (the `main` baseline).
- **Post token set** — the **working-tree** `packages/tokens/dist/products/css/tokens.css`.
- **Changesets** — `.changeset/*.md`.
- **Source JSON** — `packages/tokens/src/**/*.json` (excluding `packages/tokens/src/carbon-extracted/**`, which is unchanged and irrelevant), read from `main` and the working tree.

### Processing

1. **Extract names** (values ignored) from pre and post `tokens.css` into two name sets.
2. **Classify** by name: `unchanged` (in both), `removed` (pre only), `added` (post only).
3. **Extract S1** rename chains from changesets; **extract S2** structural renames from the source-JSON diff; **detect systematic patterns** and, where relevant, ask the maintainer to confirm.
4. **Propose** an `after` for each `removed` token by combining S1 → S2 → S3; attach the agreeing `signals` and a `confidence`.
5. **Flag conflicts** (e.g. two different candidates for the same `before`) explicitly.

### Outputs (`reports/<codebase>/`, e.g. `reports/hds/`)

- **`token-map.generated.json`** — the proposed map and **source-of-truth** structure: an object with **three arrays** keyed by the pre/post name classification, which is easy to read, diff, and hand-edit, and which drops the need for any per-entry `match` flag:

  ```jsonc
  {
    // present in both pre and post — names only; Phase B ignores these entirely
    "unchanged": [
      "--hds-border-radius-medium",
      "--hds-spacing-4"
    ],
    // pre-only tokens — the ONLY array Phase B acts on; `after` is the proposed new
    // name (a post token) or `null` when a removed token has no successor (manual decision)
    "removed": [
      { "before": "--hds-color-border-primary", "after": "--hds-border-color-primary" },
      { "before": "--hds-color-foreground-strong", "after": "--hds-foreground-color-strong" },
      { "before": "--hds-some-removed-token", "after": null }
    ],
    // post-only names (inventory / reference) — includes rename targets and brand-new tokens; not migration targets
    "added": [
      "--hds-border-color-primary",
      "--hds-foreground-color-strong",
      "--hds-brand-new-token"
    ]
  }
  ```

  Together, `unchanged` + every `removed[].before` account for **every pre token**, so the map doubles as a complete inventory of what happened to each old token. Phase B only ever reads `removed`; `unchanged` and `added` are there for human reference and diffing.
- **`token-diff.md`** — the detailed human report: summary counts, renames grouped by confidence (high → low), source-of-signal annotations (which of S1/S2/S3 agreed), detected systematic patterns awaiting confirmation, unmatched `removed` / `added` tables, and conflicts. This is where a reviewer goes for depth; the JSON stays minimal and editable.

### Confirming the map

The human reviews `token-diff.md`, edits/completes `token-map.generated.json`, and saves it as **`token-map.json`** — the confirmed source of truth consumed by Phase B. Phase A always (re)writes `token-map.generated.json` (safe to overwrite on every run) and never overwrites the human-owned `token-map.json`.

### CLI

```bash
node copilot-plans/full-tokens-replacement/tooling/diff-tokens.mjs
```

## 6. Phase B — `migrate-tokens.mjs` (applier) + verification

> **Depends on the human-confirmed `token-map.json`.** Phase B **never** applies the raw generated proposal — only the reviewed, confirmed map.

### Inputs

Again, the only CLI argument is `--config <path>`. The config points at the confirmed `token-map.json` and the scan roots/extensions/excludes (HDS defaults: `packages/components/src`, `showcase/app`, `showcase/tests`, `website/app`, `website/docs`; extensions `scss, css, gts, hbs, ts, js`).

### Processing

1. Load the confirmed map; iterate the **`removed`** array and build a lookup of `before → after` for entries whose `after` is non-null. (The `unchanged` and `added` arrays are ignored outright — nothing to rewrite there.)
2. Walk configured roots (reusing the verify core's `collectFiles` / `walk` / glob excluder / `stripComments`).
3. Replace `var(--old)` and bare `--old` custom-property references with `--new`. **Mechanical, idempotent, position-preserving.**
4. **Skip interpolated** dynamic names (`--hds-…#{…}` / `--hds-…${…}`) and **flag** them for manual review (cannot be statically rewritten).
5. **Removed tokens with no confirmed `after`** (`after: null`) → insert a `🚧 TODO [HDS-TOKEN-CARBONIZATION]` marker comment at the usage site instead of editing (mirrors the convention in `copilot-plans/hds-tokens-replacement/`), and report each occurrence.
6. When `dryRun: true` in config, print planned edits + counts and write no source files (report is still written).

### Outputs

Written to `reportDir` (default `reports/<codebase>/` — the same folder as the map).

- **`token-migration.md`** — the **human-readable** report: replacements per token, files changed, skipped interpolations, and unmapped-token TODO locations. This is what a reviewer reads.
- **`token-migration.json`** — the **machine-readable** twin of the same data (structured, stable shape), for programmatic use: CI assertions (e.g. “fail if any `after:null` usages remain”), diffing runs against each other, driving follow-up scripts, and keeping an audit trail. It is **not** consumed by Phase B itself — it's an output artifact for downstream tooling/automation.

### Verification pass

Re-scan the roots (reusing the `verify-tokens` scan) and confirm **zero stale pre-only token names** remain (excluding intentionally-flagged / interpolated cases). Report any residuals.

### CLI

```bash
node copilot-plans/full-tokens-replacement/tooling/migrate-tokens.mjs \
  --config copilot-plans/full-tokens-replacement/tooling/config/migrate.hds.config.json
```

## 8. Decisions & default choices

1. **Names only, values ignored** — the migration rewrites token names; token values (even where they changed during carbonization) are irrelevant.
2. **No value-match signal** — intentionally excluded as too weak (false positives/negatives); see §4.
3. **Config-only CLI (Phase B)** — Phase B's sole CLI argument is `--config`; all its inputs live in the config file. Phase A takes **no config and no arguments** (fixed one-off paths hardcoded in the script).
4. **Phase A is one-off; Phase B is portable** — Phase A runs once here with fixed HDS inputs (pre `main`, post working tree); only Phase B and the token map travel to other repos.
5. **Source-of-truth token map** — a simple, hand-editable JSON object of three arrays (`unchanged` / `removed` / `added`) doubles as tooling input, human reference, and documentation source; Phase B reads only `removed`.
6. **Sass `$hds-*` / `$token-*` variables** — CSS custom properties only by default; a configurable `sassPrefixes` option enables Sass-variable coverage per repo when needed.
7. **Two-phase separation** — Phase B consumes **only** the human-confirmed `token-map.json`; the generated proposal is never applied automatically.
8. **Scope** — `--hds-*` token usages in source; excludes `dist/**`, `node_modules`, `*.map`, `*.md` (as edit targets), compiled showcase assets, and `themed-tokens/**`.

## 9. Verification strategy

1. **Phase A sanity check** — run the diff (pre `main` vs post working tree); open `token-diff.md`; confirm changeset chains resolve to real pre/post endpoints; spot-check a few renames against the source JSON under `packages/tokens/src/products`; confirm any detected systematic patterns with the maintainer.
2. **Phase B dry-run** — set `dryRun: true`; inspect planned edits and counts before touching files.
3. **Apply on a clean checkout** — review the Git diff; run `pnpm lint` / `pnpm lint:fix`. **Do not run the test suite from the agent** — ask the maintainer to run tests and report back.
4. **Idempotency** — re-running from a clean checkout must produce a **byte-for-byte identical diff**.
5. **Post-apply verify pass** — reports **0 stale pre-token names** (excluding intentionally flagged / interpolated usages).

## 10. Open questions / risks

- **Multi-step rename chains** — the biggest inference subtlety: a changeset's `before → after` may be an intermediate step. Endpoints must be composed and reconciled against the actual pre/post name sets, or the map will point at a name that no longer exists.
- **Changeset coverage vs. completeness** — if some carbonization renames are only in placeholder changesets, S2–S3 must fill the gap; those land in low confidence and need manual confirmation. The `token-diff.md` unmatched tables make them visible.
- **Broad changeset scope** — changesets cover the whole carbonization effort (mechanical prefix swaps and non-mechanical refactors) and sometimes reference component-internal `--hds-var-*` variables absent from `tokens.css`; scoping composed endpoints against the actual pre/post name delta filters out anything that isn't part of this run's token migration, but review is still advised.
- **Systematic patterns need confirmation** — bulk pattern application is powerful but must be maintainer-confirmed to avoid over-applying a rule to tokens that were actually handled differently.
- **Indirect (non-`--`) coupling** — code that couples to a bare token `name` string (JS/TS reading the tokens JSON) won't be caught by CSS-usage scanning; these are flagged with the TODO marker for manual review, consistent with the replacement plan.
